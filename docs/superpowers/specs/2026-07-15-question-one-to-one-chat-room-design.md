# 질문 1:1 채팅방 설계

## 목표

질문을 계기로 두 사용자가 1:1 채팅을 시작할 수 있게 한다.

- 흐름: 질문 → 답변 → 채택 → 사용자가 질문 채팅방 생성
- 방 타입: `question`
- 방 제목: 질문 제목
- 방 식별 정보: `question_id`
- 자동 생성 이벤트는 두지 않는다.

질문 채팅은 기존 1:1 채팅의 또 다른 타입이다. 별도 채팅 상태 머신이나 계약 계층을 만들지 않는다.

## 생성 규칙

`POST /api/v1/chat/rooms/question`은 `questionId`와 `targetUserId`를 받는다.

생성 시 다음만 검증한다.

1. 질문이 존재하고 삭제되지 않았는가
2. 요청자와 대상 사용자가 활성 사용자인가
3. 요청자와 대상이 서로 다른가
4. 두 사용자 사이에 차단 관계가 없는가

친구 관계, 질문 작성자 여부, 답변 작성자 여부, 답변 채택 여부는 생성 조건이 아니다.

같은 질문과 같은 두 사용자 조합은 기존 `room_key`를 사용해 같은 방을 반환한다. 사용자 순서는 방 식별에 영향을 주지 않는다.

## 나가기 규칙

채팅 화면의 뒤로가기는 서버 API를 호출하지 않으며 방 상태를 바꾸지 않는다.

실제 1:1 방 나가기는 다음처럼 처리한다.

- `chat_members.left_at`을 기록한다.
- 해당 사용자의 채팅 목록에서 방을 제거한다.
- 나간 사용자에게 그 시점까지의 메시지를 숨긴다.
- 기존 방을 다시 열거나 상대가 새 메시지를 보내 멤버십이 복구되면 이후 메시지만 보인다.
- 상대 사용자의 메시지 이력에는 영향이 없다.

이를 위해 `chat_members.visible_after_message_id`에 나가기 시점의 방 최대 메시지 ID를 저장한다. 사용자 화면용 메시지 이력, 안 읽은 수, 마지막 메시지는 모두 다음 조건을 적용한다.

```sql
message_id > visible_after_message_id
```

메시지가 없는 방의 기준값은 `0`이다. 신고 문맥 조회처럼 운영 목적의 조회는 이 사용자 화면 가시성 규칙과 분리한다.

## 데이터베이스 변경

```sql
visible_after_message_id BIGINT NOT NULL DEFAULT 0
```

음수 값은 허용하지 않는다.

기존 버전에서 `direct` 또는 `question` 방을 나가 `left_at`이 남은 멤버는 배포 마이그레이션에서 다음과 같이 보정한다.

- `visible_after_message_id`: 해당 방의 현재 최대 메시지 ID
- `left_at`: 기존 값을 유지

따라서 기존에 나간 1:1 방은 목록에서 계속 제외되고, 이후 멤버십이 복구되더라도 과거 메시지는 보이지 않는다. 그룹 방의 이탈 의미는 기존대로 유지한다.

## 응답

질문 방 생성·목록·상세 응답은 기존 방 응답 구조를 사용하면서 다음 값을 제공한다.

- `roomType = question`
- `questionId`
- `questionTitle`

## 검증 범위

1. 친구나 질문/답변 당사자가 아니어도 질문 방을 생성할 수 있다.
2. 차단·본인 대상·삭제된 질문·비활성 사용자는 거부한다.
3. 같은 질문과 사용자 조합은 같은 방을 반환한다.
4. 뒤로가기는 상태를 바꾸지 않고 방을 목록에 유지한다.
5. 실제 나가기는 `left_at`을 기록하고 방을 목록에서 제거한다.
6. 나간 사용자가 새 메시지로 다시 활성화되면 기존 이력은 숨고 새 메시지만 보인다.
7. 상대방의 기존 이력은 유지된다.
8. 레거시 스키마에 마이그레이션을 적용하면 기존 1:1 이탈 데이터가 새 규칙으로 보정된다.
